2 Workflow shop-qa-ui — co dokładnie się dzieje

shop-qa-ui to agent LLM działający nad kodem 8 repozytoriów sklepu. Prowadzi użytkownika przez 4-krokowy workflow (stepper w UI): System Ready → Analyze → Piaskownica → PR. Poniżej dokładny przebieg każdego kroku wraz z funkcjami, które go realizują.

manifest.yaml → indeks (CodeChunk[]) → run_qa (2× LLM) → rekomendacja → plan_change (mapa repo) → generate_file_change/new_file → verify_completeness → open_pr_for_files (git worktree + gh) → pr_checks (bramka) → merge_pr (każde wywołanie LLM → _emit_token_metrics → shop-token-metrics)

Konfiguracja wejściowa

Repozytoria
manifest.yaml → wpisy indexable: true (8 repo), ścieżki względem workspace_root (nadpisywalne SHOP_REPOS_DIR)
Klucz LLM
OPENROUTER_API_KEY (przez .env.docker w kontenerze / .env na hoście). Pusty/błędny → 401 i błąd w odpowiedzi.
Model / rozumowanie
OPENROUTER_MODEL=z-ai/glm-5.2, reasoning: effort=high, usage.include=true
Telemetria
TOKEN_METRICS_URL → po każdym wywołaniu LLM _emit_token_metrics() wysyła usage (best-effort, nie blokuje)
Krok 1

System Ready — indeksowanie

W panelu bocznym wybierasz aplikację (z manifest.yaml) i klikasz „⚡ Indeksuj aplikację". To operacja czysto lokalna i statyczna — bez LLM i bez sieci:

  • ingest_app(repos) → dla każdego repo ingest_repo(path): rekurencyjny rglob po .py/.java/.js/.jsx/.ts/.tsx/.mjs, pomijając _SKIP_DIRS (build/ .gradle/ node_modules/ .venv/ .git/ bin/ dist/ __pycache__/).
  • Parsery per język: Python — AST (FunctionDef/ClassDef); Java — regex + liczenie głębokości {} (typy + metody/konstruktory); JS/TS — segmentacja deklaracji top-level.
  • Wynik: lista CodeChunk (repo, file_path, symbol, start_line, end_line, source) → zapis do st.session_state.chunks + mapa repo_paths.

Ekran „System Ready" pokazuje liczbę zaindeksowanych symboli i przycisk „Przejdź dalej →".

Charakter indeksu: retrieval jest słownikowy (nie embeddingi), indeks jest ulotny (żyje w session_state, znika po odświeżeniu / „Nowym wątku"). Wiedza dziedzinowa (saga, zdarzenia) nie pochodzi z indeksu — jest w stałej _SHOP_FACTS wstrzykiwanej do każdego promptu.
Krok 2

Analyze — pytanie, odpowiedź i rekomendacja

Wpisujesz pytanie (np. „usuń z shop-ui możliwość logowania i wszystko z tym związane"). Klik „Zapytaj ➜" wywołuje run_qa(question, chunks):

  1. Retrieval: keyword_search(chunks, question, top_k=5) — przecięcie tokenów pytania i symbol+source chunka; brak trafień → zwraca całość (fallback).
  2. Budowa kontekstu: _build_context() skleja 5 chunków z nagłówkami REPO/PLIK · FUNKCJA · LINIE. Prompt = _SHOP_FACTS + KOD + PYTANIE.
  3. Wywołanie LLM #1 (_TECH_SYSTEM) → odpowiedź techniczna wyłącznie na podstawie dostarczonego kodu.
  4. Wywołanie LLM #2 (_BIZ_SYSTEM, JSON) → business_context (czas dev/test, impact, ryzyko, zależności), feasibility (werdykt + serwisy + pliki), test_plan, proposals[] oraz recommended_index/recommended_reason.

Odpowiedź renderowana w 4 zakładkach: Podsumowanie · Wykonalność · Plan testów · Źródła (z cytatami repo/plik:linia i podglądem kodu). LLM wskazuje jedną rekomendację + rozważane alternatywy.

  • Doprecyzowanie: możesz dopisać uwagi („✍️ Doprecyzuj") → pytanie jest wzbogacane o twarde wymagania i run_qa jest wołane ponownie (nowa rekomendacja).
  • „✅ Akceptuj rekomendację i przejdź dalej" → do session_state trafiają sandbox_accepted_proposals, sandbox_question, sandbox_chunks, a workflow przechodzi do Piaskownicy.
Krok 3

Piaskownica — planowanie, generowanie, weryfikacja, PR

Dotyczy plików serwisowych (nie-.py). Przebiega etapami:

3a. Planowanie zmiany w całej aplikacji

build_repo_map(chunks) tworzy konspekt całej aplikacji (repo → plik → symbole). plan_change(question, repo_map, accepted_proposals) pyta LLM, które pliki zmienić/utworzyć — zwraca JSON listę {repo, path, action: modify|create, reason}. To pozwala działać cross-repo, a nie tylko na top-5 z retrievalu. Dla modify wczytywany jest oryginał pliku; brak pliku → create.

3b. Generowanie treści per plik

  • modify → generate_file_change(question, proposals, path, original) — pełna nowa treść pliku.
  • create → generate_new_file(question, proposals, repo, path).
  • compute_diff(original, new) (difflib) → podgląd diff w UI. Pusta treść z modelu = brak zmiany (celowo, żeby nie wygenerować diffu kasującego cały plik).

3c. Kontrola kompletności (agent-recenzent)

Licznik „wygenerowano N/M plików" + „♻️ Ponów puste". verify_completeness(question, proposals, repo_map, generated_files) — drugi agent ocenia, czy zestaw plików wystarcza end-to-end; zwraca {complete, notes, missing[]}. Brakujące pliki można jednym kliknięciem dodać do planu i wygenerować.

3d. Wystawienie Pull Requestów

Pliki są grupowane per repo → jeden PR na repo. open_pr_for_files(files, title, body, "ai-bot-playground/<repo>", local_repo):

git fetch origin main → git worktree add (świeży origin/main) → zapis plików → git add/commit → git push -u origin <branch> → gh pr create --base main --head <branch>
Testy lokalne nie blokują wystawienia PR — właściwa walidacja dzieje się w CI (preprod-gate). Opcjonalnie run_service_tests() uruchamia gradlew test w izolowanym worktree (tylko serwisy Gradle). Cała ścieżka operuje na lokalnych klonach repo + gh — nie dotyka uruchomionego stacku compose.

Wynik (URL/branch/success per repo) trafia do qa_multi_prs i workflow przechodzi do kroku PR.

Krok 4

PR — status bramki i merge (human-in-the-loop)

Panel PR pokazuje otwarte Pull Requesty i odświeża status bramki co 15 s (@st.fragment(run_every=15)):

  • pr_checks(repo_slug, branch) (przez gh pr checks --json) → agregacja stanów: ✅ success · ⏳ pending · ❌ failure · ⚠️ nieotwarte.
  • Przy failure: lista nieudanych checków z linkami do logów + pr_failure_summary() (ogon logu CI) + komunikat „deweloper powinien zajrzeć i naprawić".
  • Ścieżka pojedynczego serwisu Java: po zielonej bramce preprod-gate UI podaje komendę kubectl port-forward do podglądu na preprod, a merge jest za potwierdzeniem człowieka: checkbox „potwierdzam, że działa" → merge_pr() (gh pr merge --squash --delete-branch).
Pętla domknięta: zmiana wygenerowana przez agenta jest wdrażana na realny klaster kind-preprod przez bramkę, człowiek weryfikuje efekt i dopiero wtedy merguje do main. Żaden krok nie omija bramki jakości.

Przekrojowo — telemetria tokenów

Każde wywołanie LLM (run_qa, plan_change, generate_*, verify_completeness) przechodzi przez _call_openrouter(), który po odpowiedzi wywołuje _emit_token_metrics() → POST /api/usage do shop-token-metrics. Micrometer wystawia liczniki (llm_tokens_total, llm_requests_total, llm_cost_usd_total), Prometheus je scrapuje, a Grafana rysuje dashboard „LLM Token Usage". Indeksowanie realistycznego, wielorepowego kodu jest więc krokiem 0, który generuje mierzalny ruch LLM obserwowany przez cały stack.
